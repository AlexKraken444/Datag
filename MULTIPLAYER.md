# Мультиплеер и безопасность

## Транспорт

- **WebRTC DataChannel** (peer-to-peer, без серверов).
- Рукопожатие — через **публичный broker PeerJS** (`0.peerjs.com`, бесплатный, без аутентификации). Broker нужен только для обмена SDP/ICE; после соединения весь трафик идёт напрямую между браузерами.
- **STUN:** `stun.l.google.com`, `stun1.l.google.com`.
- **TURN (fallback):** публичный OpenRelay (Metered) — для строгих NAT-конфигураций. Можно заменить на свой в `game/host/ice.ts`.

## Топология

```
              ┌────────────┐
              │   HOST     │  ← создатель комнаты
              │  browser   │  Peer ID = код комнаты ("ABCD23")
              └─────┬──────┘
                    │ WebRTC DataChannel (один на игрока)
       ┌────────────┼─────────────┐
       │            │             │
   ┌───▼───┐    ┌───▼───┐     ┌───▼───┐
   │ Peer  │    │ Peer  │     │ Peer  │
   └───────┘    └───────┘     └───────┘
   joiners (до 3-х)
```

Звезда вокруг хоста. Игроки не общаются друг с другом напрямую — всё через хост.

## Контракт сообщений

См. `types/messages.ts`. Кратко (TypeScript discriminated unions):

### Peer → Host (`PeerToHost`)
| Сообщение  | Когда                                 |
| ---------- | ------------------------------------- |
| `hello`    | сразу после открытия канала           |
| `select`   | выбор команды/роли в лобби            |
| `ready`    | установка/снятие готовности           |
| `start`    | хост дёргает старт (от других игнор)  |
| `chat`     | сообщение в чат                       |
| `input`    | геймпэйновый ввод 30 раз/сек          |

### Host → Peer (`HostToPeer`)
| Сообщение     | Когда                                              |
| ------------- | -------------------------------------------------- |
| `welcome`     | хост принял подключение, шлёт твой LobbyPlayer     |
| `room`        | полная картина лобби (после каждого изменения)     |
| `chat`        | новое сообщение чата                               |
| `match_start` | начало матча                                       |
| `snapshot`    | состояние арены (≈30 раз/сек)                      |
| `round_end`   | хит / ничья / бонус                                |
| `match_end`   | финальный счёт + статистика                        |
| `kicked`      | хост выкинул (нет места, матч идёт)                |
| `error`       | ошибка                                             |

## Жизненный цикл

```
[создание]    HostController.init(nickname)
              ├─ new Peer(roomCode)
              └─ peer.on("connection", conn => ...)

[join]        PeerClient.connect()
              ├─ new Peer() → peer.connect(roomCode)
              └─ datachannel.open → send("hello", nickname)

[лобби]       выбор команды/роли, чат, готовность.
              4 готовых → host автоматически запускает HostMatch.

[матч]        HostMatch.start() — HostLoop 60Hz
              ├─ принимает input от пиров (и от себя через localInput)
              ├─ симулирует Player/Light/Shadow/Collision
              ├─ начисляет Score
              └─ broadcast(snapshot) 30Hz через все DataChannel'ы.

[конец]       match_end → host POST'ит результат в /api/match-result (best-effort)
```

## Безопасность / античит

**Архитектура — host-authoritative.** Хост (создатель комнаты) — источник правды. Все остальные клиенты только показывают то, что хост им шлёт.

Что **по-прежнему защищено**:

| Угроза                                    | Защита                                          |
| ----------------------------------------- | ----------------------------------------------- |
| Самонатягивание очков пиром               | Пиры не считают хиты — это делает HostMatch     |
| Телепорт пира (move = бесконечная скорость) | `PlayerSystem` клампит позицию по `TICK.DT`     |
| Lighter за пределами зоны                 | `LightSystem` зажимает по AABB                  |
| Спам input                                | Хост обрабатывает не чаще одного input в тик    |
| Подделка ника                             | Поле `nickname` валидируется регуляркой         |
| Подсадка чужого peer ID                   | PeerJS broker не даёт коллизий                  |

Что **не** защищено и зависит от хоста:

- Сам хост может читерить (его браузер исполняет симуляцию).
- При краше браузера хоста матч умирает.

Это компромисс serverless-деплоя: либо отдельный игровой сервер (Railway/Render/VPS), либо доверие к хосту. Для матчей с друзьями этого хватает.

POST `/api/match-result` тоже не доверенный (любой может слать). Если планируете публичный лидерборд — добавьте подписи nonce + проверку на стороне Vercel-роута.

## Производительность

- Snapshot ≈600 байт × 30Hz = ~18 KB/sec на пира.
- Latency: WebRTC обычно 20–80 ms между друзьями, без серверного хопа.
- Host CPU: симуляция 4 игроков + 2 теней + столкновения = доли процента.

## Reconnect

- Если пир обрывает связь во время матча — `connected=false`, слот за ним остаётся.
- При повторном `hello` с тем же ником хост вернёт его на старый слот и сразу дошлёт snapshot.
- Если **хост** обрывается — матч теряется (нет резервного хоста). При необходимости можно реализовать host failover через WebRTC election; не входит в MVP.
